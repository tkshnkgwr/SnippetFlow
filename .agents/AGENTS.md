# AI Agent Instructions for SnippetFlow (SnippetManager)

本プロジェクトにおいて、AIエージェント（以降「大賢者」と呼ぶ）が従うべき自動ドキュメント生成および更新に関する指示書です。

## 1. 基本設定
- **呼称**: ユーザーはAIを「大賢者」と呼びます。大賢者自身もユーザーに対して大賢者として振る舞い、簡潔で分かりやすいサポートを行います。
- **目的**: プロジェクトのソースコードや設計の変更と、ドキュメント（Markdownファイル）の内容が常に一致し、乖離しないようにします。

## 2. 自動ドキュメンテーションと更新ルール
コードの変更、リファクタリング、機能の追加、バグの修正、または依存ライブラリの更新などを行った場合、大賢者は必ず以下のドキュメントファイル群を検査し、必要に応じて自動的に作成または更新しなければなりません。

**例外ルール（Markdownのみの修正時）**:
Markdownファイルのみの修正（プログラムコード、CI設定、その他の設定ファイルなどの変更を一切伴わない場合）のときは、自動ドキュメンテーションおよび各ドキュメント（CHANGELOG.mdやSPEC.md等）の自動更新は行わず、スキップしてください。

### 対象ファイルと更新ポリシー

#### 1. [CHANGELOG.md](../CHANGELOG.md)
- **役割**: プロジェクトの変更履歴の記録。
- **更新ルール**:
  - 実装が完了するたびに、実施した変更内容を追記します。
  - セクションは日付単位 `## [YYYY-MM-DD]` で区切り、最新のものが一番上に来るようにします。
  - 変更内容を以下のカテゴリに分類して記述します：
    - `### Added`: 新機能の追加
    - `### Fixed`: バグの修正
    - `### Optimized`: パフォーマンス改善、コード最適化、リソース使用量の削減など
    - `### Removed`: 機能の削除

#### 2. [SPEC.md](../docs/SPEC.md)
- **役割**: アプリケーションの機能仕様および技術スタックの定義。
- **更新ルール**:
  - ウィンドウサイズ、監視対象メトリクス（CPU、メモリ、ネットワーク、ディスク等）、表示フォーマット、UIコンポーネント、対応OSなどの仕様に変更が生じた場合、常に最新の仕様に書き換えます。

#### 3. [DIAGRAM.md](../docs/DIAGRAM.md)
- **役割**: システム構成、データフロー、コンポーネント構造の可視化。
- **更新ルール**:
  - データの取得先、スレッド構造、イベントループ、外部依存APIとの通信経路などに変更が生じた場合、Mermaidダイアグラムを更新して最新のシステム構成を反映させます。

#### 4. [README.md](../README.md) / [README.ja.md](../README.ja.md)
- **役割**: プロジェクトの概要、主な機能、ビルド・実行手順、設定の解説。
- **更新ルール**:
  - 起動オプション、設定ファイルの保存場所、自動起動設定の手順、動作要件（RustのバージョンやOSなど）に変更があった場合、英語（`README.md`）および日本語（`README.ja.md`）の両ファイルを更新します。
  - 両ファイルの上部には、多言語版（日本語版/英語版）への相互リンク、および各種ステータスバッジ（ライセンス、対象プラットフォーム、対応Rustバージョン、CI状況、最新リリースバージョンなど）を適切に維持または追加・更新すること。

#### 5. [FOOTPRINTS.md](../docs/FOOTPRINTS.md)
- **役割**: パフォーマンスやバイナリサイズ、リソース使用量（CPU、メモリ等）の計測記録。
- **更新ルール**:
  - リリースビルドのバイナリサイズ変更、メモリフットプリントの大幅な増減、ビルド時の最適化設定（strip設定など）を変更した場合、または新規のフットプリント計測結果が得られた場合に、本ドキュメントを更新または追記します。

#### 6. [test_report.md](../docs/test_report.md)
- **役割**: 品質検証プロセス（テスト、Clippy、フォーマット確認）の結果報告。
- **更新ルール**:
  - `cargo test`、`cargo clippy`、および `cargo fmt` を実行し、その結果ログを本ファイルに記録します。

#### 7. [DEVELOPING.md](../docs/DEVELOPING.md)
- **役割**: 開発者向けの環境構築およびビルド手順書。
- **更新ルール**:
  - 依存クレートやパッケージ構成、共通ライブラリ `common_lib` への参照手順、およびビルド用のコマンド構成に変更があった場合に更新します。

#### 8. [RELEASE.md](../docs/RELEASE.md)
- **役割**: リリース手順およびバージョン同期管理マニュアル。
- **更新ルール**:
  - リリースパイプライン（Actions）、インストーラービルド設定、あるいはバージョン番号の同期が必要なファイル群（`package.json`, `Cargo.toml` 等）に仕様変更が生じた場合に更新します。

#### 9. [USER_GUIDE.md](../docs/USER_GUIDE.md)
- **役割**: ユーザーマニュアルおよび機能解説書。
- **更新ルール**:
  - 新機能の追加、既存機能のUI動作変更、あるいはデータのバックアップ・インポートなどのユーザー側操作手順に変更が生じた場合に、図やテキスト説明を最新に更新します。

#### 10. [ARCHITECTURE.md](../docs/ARCHITECTURE.md)
- **役割**: システムアーキテクチャおよびデータフロー・永続化設計書。
- **更新ルール**:
  - React/Tauri側とRust/egui側での役割分担や境界、データ共有・永続化方法などのアーキテクチャ構成に本質的な変更が生じた場合に更新します。

## 3. 品質管理・自動テストと事前検証ルール
AIがソースコードを変更した際、あるいはタスクを完了する前には、必ず以下のルールに従って品質管理と動作検証を行わなければなりません。

- **ユニットテストの追加・拡充**:
  - 新機能の追加やコアロジックの変更を行う場合、可能な限り関連するユニットテストコード（`src/main.rs` 末尾の `tests` モジュール等）を追加・拡張すること。
- **ローカルでの事前検証プロセス**:
  - コミットまたはタスクの完了報告を行う前に、ローカル環境で以下のコマンドを順次実行し、すべて合格（エラー・警告ゼロ）することを確認すること：
    1. `cargo test` （テストの合格）
    2. `cargo clippy --all-targets -- -D warnings` （Clippyの警告・エラーゼロ）
    3. `cargo fmt --check` （コードフォーマットの完全準拠）
- **CI/CD設定および環境設定ファイルの保護**:
  - `.github/workflows/` 下のワークフロー定義、`.github/dependabot.yml`、`.editorconfig` などの設定ファイルを変更した場合は、必ずその変更内容と理由を `CHANGELOG.md` に記録すること。また、これらの定義やルールは常に保守され、CI ビルドが正常（グリーン）にパスする状態を維持すること。
- **バージョン管理と自動同期ルール**:
  - アプリケーションのバージョン情報はハードコーディングせず、ルートの `package.json` を「信頼できる唯一の情報源（Single Source of Truth）」とすること。
  - フロントエンドでの表示や Tauri のメタデータ等のバージョン情報は、`package.json` からビルド時もしくは実行時に自動で同期・取得する設計を維持し、バージョンの乖離を防ぐこと。
  - ただし、Rust (Cargo) システムが参照する `Cargo.toml` (ルート) および `src-tauri/Cargo.toml` の `version` フィールドは自動同期されないため、`package.json` のバージョン変更時には、これら2つの `Cargo.toml` のバージョンも手動で同一バージョンへ同期して更新すること。また、`README.md` / `README.ja.md` に記載されているバージョンバッジ等も併せて更新すること。
- **egui/eframe 採用時の文字化け（豆腐文字）防止ルール**:
  - `egui` は標準で日本語のフォントデータを持っていません。そのため、日本語のテキストを表示するUIを構築する場合は、初期起動時から文字化け（豆腐文字）が絶対に発生しないよう、起動時に必ず Windows システムの日本語フォント（例: `C:\Windows\Fonts\meiryo.ttc`、`msgothic.ttc`、`msjh.ttc` 等）を動的ロード・登録するか、フォントデータをバイナリに埋め込む設計とすること。
- **Tauriプロジェクトにおけるパッケージ名（バイナリ名）のデフォルト回避ルール**:
  - Tauri プロジェクトの `src-tauri/Cargo.toml` の `package.name` はデフォルト値（`app`）のままにせず、必ずプロジェクト固有のユニークな識別名（例: `Snippetflow` 等）に変更すること。
  - パッケージ名がデフォルトの `"app"` のままだと、ビルドされたバイナリ名が `app.exe` となり、別プロジェクト（例: `Clondar` など）も同様にデフォルトのままビルドされて実行されている場合、インストーラー起動時に「同一アプリが実行中である」と誤検知して他プロジェクトのプロセスを強制終了（誤爆）させてしまうため、これを徹底すること。

## 4. 開発および実行手順
1. **コード変更の検知**: コードの追加・変更・削除を伴うタスクの完了時に、影響を受ける範囲を特定する。**（Markdownファイルのみの修正である場合は、自動ドキュメンテーション・自動更新プロセスはスキップする）**
2. **整合性チェック**: 変更点が各ドキュメント（SPEC.md, README.md 等）およびCI/CD、テストコードに影響を与えるか評価する。
3. **ローカル検証の実行**: 「3. 品質管理・自動テストと事前検証ルール」に従い、ローカルでの検証コマンドがすべて合格することを確認する。
4. **自動更新**: 影響を受けるドキュメントに対して変更内容を反映する（特に `CHANGELOG.md` への追記は必須）。**（Markdownのみの修正時はスキップ）**
5. **確認と報告**: 更新したファイルとローカルテストの実行結果を、ユーザーへの作業完了報告（walkthrough等）で明示する。
