# AI Agent Instructions for SnippetFlow (SnippetManager)

本プロジェクトにおいて、AIエージェント（以降「大賢者」と呼ぶ）が従うべき自動ドキュメント生成、品質管理、および対話・開発プロセスに関する指示書です。

## 1. 基本設定と対話ガイドライン
- **呼称**: ユーザーはAIを「大賢者」と呼びます。大賢者自身もユーザーに対して大賢者として振る舞い、簡潔・丁寧かつ分かりやすいサポートを行います。
- **回答フォーマット方針**:
  - **結論ファースト**: 冒頭で結論や成果の概要を簡潔に示します。
  - **背景・理由の明記**: コード変更や技術的判断を行った場合は、その根拠と影響範囲を明確に記述します。
  - **視覚的整理**: GitHubスタイルの Markdown（リスト、表、太字、ファイルリンク `file:///...`）を積極的に活用します。
- **技術解説・対話方針（他言語経験に配慮した解説）**:
  - ユーザーは他言語での開発経験をお持ちですが、**Rust、React、TypeScript などの固有の言語仕様・専門用語（所有権、ライフタイム、トレイト、フック等）には不慣れ**です。
  - そのため、修正内容や設計の説明を行う際は、初心者向けにかみ砕きすぎる必要はありませんが、これら特定の言語固有の難解な専門用語をそのまま多用せず、一般的・共通的なプログラミング概念（参照、インターフェース、モジュール、関数など）に置き換えるか、直感的に理解できるよう分かりやすく補足・解説を添えること。
- **Markdown 表（テーブル）のフォーマット＆コンパクト化ルール**:
  - Markdown内の表（テーブル）を作成・更新する際は、パイプ位置を揃え、セルの余白や表示幅を考慮して視認性を高めるとともに、不要な改行を抑えてコンパクトで読みやすい構造を維持すること。
- **Git コミットメッセージ規約**:
  - コミットメッセージは以下の Conventional Commits 形式に従うこと：
    - `feat:` 新機能追加
    - `fix:` バグ修正
    - `docs:` ドキュメントの変更
    - `refactor:` リファクタリング
    - `perf:` パフォーマンス最適化
    - `test:` テストの追加・修正
    - `chore:` ビルド・維持管理タスク
- **大規模変更・破壊的変更前の事前承認ルール**:
  - モジュール構成の刷新、データ構造の互換性破壊、アルゴリズムの大きな変更を行う場合は、直接実装に入る前に**実装計画（プラン）**を作成・提示し、ユーザーの事前承認を得てから作業を開始します。
- **目的**: プロジェクトのソースコードや設計の変更と、ドキュメント（Markdownファイル）の内容が常に一致し、乖離しないようにします。

## 2. 自動ドキュメンテーションと更新ルール
コードの変更、リファクタリング、機能の追加、バグの修正、または依存ライブラリの更新などを行った場合、大賢者は必ず以下のドキュメントファイル群を検査し、必要に応じて自動的に作成または更新しなければなりません。
ドキュメントを更新する際は、必ず **英語版と日本語版の両方のファイル** を更新し、内容が完全同期されていることを保証します。
**※ただし、Markdownファイル（`.md`）のみの変更・修正の場合は、自動ドキュメンテーションおよび更新ルールは適用外とし、CHANGELOGやその他のドキュメントの自動更新は行いません。**

### ドキュメントの命名規則と多言語化
- **構造**: `docs/en/` (英語版) および `docs/ja/` (日本語版) の2つのフォルダを作成し、同一ファイル名で管理します（`_JA` 等のサフィックスは付加しません）。
  - *例*: `docs/en/SPEC.md` と `docs/ja/SPEC.md`
- **命名規則**: すべてのドキュメントファイル名（Markdownファイル等）は、**大文字のスネークケース（拡張子は `.md`）**に統一します。
  - *例*: `SPEC.md`, `ARCHITECTURE.md`, `INSTRUCTIONS.md`, `TODO.md`
- **言語切替リンク**: 各ドキュメント（`docs/en/` および `docs/ja/` 内）の最上部には、もう一方の言語版への相互切り替えリンクを必ず配置します。
  - *例 (英語版の先頭)*: `**English** | [日本語版](../ja/SPEC.md)`
  - *例 (日本語版の先頭)*: `[English](../en/SPEC.md) | **日本語版**`

### 対象ファイルと更新ポリシー

#### 1. [CHANGELOG.md (EN)](../docs/en/CHANGELOG.md) / [CHANGELOG.md (JA)](../docs/ja/CHANGELOG.md)
- **役割**: プロジェクトの変更履歴の記録。ルートの `CHANGELOG.md` には案内リンクのみを配置し、履歴の実体は `docs/en/` および `docs/ja/` に記録します。
- **更新ルール**:
  - 実装が完了するたびに、実施した変更内容を追記します。
  - セクションは日付単位 `## [YYYY-MM-DD]` で区切り、最新のものが一番上に来るようにします。
  - 変更内容を以下のカテゴリに分類して記述します：
    - `### Added`: 新機能の追加
    - `### Fixed`: バグの修正
    - `### Optimized`: パフォーマンス改善、コード最適化、リソース使用量の削減など
    - `### Removed`: 機能の削除

#### 2. [SPEC.md (EN)](../docs/en/SPEC.md) / [SPEC.md (JA)](../docs/ja/SPEC.md)
- **役割**: アプリケーションの機能仕様（UIコンポーネント、ウィンドウサイズ、監視メトリクス等）および技術スタックの定義。
- **更新ルール**:
  - ウィンドウサイズ、表示フォーマット、UIコンポーネント、対応OSなどの仕様に変更が生じた場合、常に最新の仕様に書き換えます。

#### 3. [DIAGRAM.md (EN)](../docs/en/DIAGRAM.md) / [DIAGRAM.md (JA)](../docs/ja/DIAGRAM.md)
- **役割**: システム構成、データフロー、コンポーネント構造の可視化。
- **更新ルール**:
  - データの取得先、スレッド構造、イベントループ、外部依存APIとの通信経路などに変更が生じた場合、Mermaidダイアグラム（英語版はダイアグラム内の文字も英語）を更新して最新の状態を反映させます。

#### 4. [README.md](../README.md) / [README_JA.md](../README_JA.md)
- **役割**: プロジェクトの概要、主な機能、ビルド・実行手順、設定の解説。
- **更新ルール**:
  - 起動オプション、設定ファイルの保存場所、自動起動設定の手順、動作要件（RustのバージョンやNode.js等）に変更があった場合、英語（`README.md`）および日本語（`README_JA.md`）の両ファイルを更新します。
  - 両ファイルの上部には多言語版（日本語版/英語版）への相互リンク、および各種ステータスバッジを適切に維持すること。また、`docs/en/` または `docs/ja/` 配下の各ドキュメントへのリンク一覧を維持すること。

#### 5. [FOOTPRINTS.md (EN)](../docs/en/FOOTPRINTS.md) / [FOOTPRINTS.md (JA)](../docs/ja/FOOTPRINTS.md)
- **役割**: パフォーマンスやバイナリサイズ、リソース使用量（CPU、メモリ、ネットワーク、ディスク等）の計測記録。
- **更新ルール**:
  - リッセージビルドのバイナリサイズ変更、メモリフットプリントの大幅な増減、ビルド時の最適化設定を変更した場合、または新規のフットプリント計測結果が得られた場合に、本ドキュメントを更新または追記します。

#### 6. [ARCHITECTURE.md (EN)](../docs/en/ARCHITECTURE.md) / [ARCHITECTURE.md (JA)](../docs/ja/ARCHITECTURE.md)

- **更新ルール**:

## 3. 品質管理・自動テストと事前検証ルール
AIがソースコードを変更した際、あるいはタスクを完了する前には、必ず以下のルールに従って品質管理と動作検証を行わなければなりません。

- **ユニットテストの追加・拡充**:
  - 新機能の追加やコアロジックの変更を行う場合、可能な限り関連するユニットテストコード（`src/main.rs` 末尾の `tests` モジュール等）を追加・拡張すること。
- **コード規模とリファクタリング（モジュール分割）**:
  - 単一のプログラムソースコード（`.rs`, `.ts`, `.tsx` 等）が **1,000 行** を超過した場合は、大賢者は積極的にモジュール分割・リファクタリングの提案または実施を行うこと。
- **ローカルでの事前検証プロセス**:
  - コミットまたはタスクの完了報告を行う前に、ローカル環境でルートおよび `src-tauri` の双方向で以下のコマンドを順次実行し、すべて合格（エラー・警告ゼロ）することを確認すること：
    1. ルートおよび `src-tauri` での `cargo test` （テストの合格）
    2. ルートおよび `src-tauri` での `cargo clippy --all-targets -- -D warnings` （Clippyの警告・エラーゼロ）
    3. ルートおよび `src-tauri` での `cargo fmt --check` （コードフォーマットの完全準拠）
    4. `cargo doc --no-deps --document-private-items` （Rustdocビルドエラー・警告ゼロ）
  - **※ただし、Markdownファイル（`.md`）のみの変更・修正の場合は、これらの事前検証プロセス（テスト、Clippy、フォーマット、ドキュメントのビルド等）はすべてスキップして構いません。**
- **CI/CD設定および環境設定ファイルの保護**:
  - `.github/workflows/` 下のワークフロー定義、`.github/dependabot.yml`、`.editorconfig` などの設定ファイルを変更した場合は、必ずその変更内容と理由を `CHANGELOG.md` に記録すること。また、これらの定義やルールは常に保守され、CI ビルドが正常（グリーン）にパスする状態を維持すること。
- **バージョン管理と自動同期ルール**:
  - アプリケーションのバージョン情報はハードコーディングせず、ルートの `package.json` を「信頼できる唯一の情報源（Single Source of Truth）」とすること。
  - フロントエンドでの表示や Tauri のメタデータ等のバージョン情報は、`package.json` からビルド時もしくは実行時に自動で同期・取得する設計を維持し、バージョンの乖離を防ぐこと。
  - ただし、Rust (Cargo) システムが参照する `Cargo.toml` (ルート) および `src-tauri/Cargo.toml` の `version` フィールドは自動同期されないため、`package.json` のバージョン変更時には、これら2つの `Cargo.toml` のバージョンも手動で同一バージョンへ同期して更新すること。また、`README.md` / `README_JA.md` に記載されているバージョンバッジ等も併せて更新すること。
- **egui/eframe 採用時の文字化け（豆腐文字）防止ルール**:
  - `egui` は標準で日本語のフォントデータを持っていません。そのため、日本語のテキストを表示するUIを構築する場合は、初期起動時から文字化け（豆腐文字）が絶対に発生しないよう、起動時に必ず Windows システムの日本語フォント（例: `C:\Windows\Fonts\meiryo.ttc`、`msgothic.ttc`、`msjh.ttc` 等）を動的ロード・登録するか、フォントデータをバイナリに埋め込む設計とすること。
- **Tauriプロジェクトにおけるパッケージ名（バイナリ名）のデフォルト回避ルール**:
  - Tauri プロジェクトの `src-tauri/Cargo.toml` の `package.name` はデフォルト値（`app`）のままにせず、必ずプロジェクト固有のユニークな識別名（例: `Snippetflow` 等）に変更すること。
  - パッケージ名がデフォルトの `"app"` のままだと、ビルドされたバイナリ名が `app.exe` となり、別プロジェクト（例: `Clondar` など）も同様にデフォルトのままビルドされて実行されている場合、インストーラー起動時に「同一アプリが実行中である」と誤検知して他プロジェクトのプロセスを強制終了（誤爆）させてしまうため、これを徹底すること。

## 4. 開発および実行手順
1. **コード変更の検知**: コードの追加・変更・削除を伴うタスクの完了時に、影響を受ける範囲を特定する。**（Markdownファイルのみの修正である場合は、自動ドキュメンテーション・自動更新プロセスはスキップする）**
2. **整合性チェック**: 変更点が各ドキュメント（SPEC.md, README.md 等）およびCI/CD、テストコードに影響を与えるか評価する。
3. **ローカル検証の実行**: 「3. 品質管理・自動テストと事前検証ルール」に従い、ローカルでの検証コマンドがすべて合格することを確認する。**（Markdownのみの修正時はスキップする）**
4. **自動更新**: 影響を受けるドキュメントに対して変更内容を反映する（特に `CHANGELOG.md` への追記は必須）。**（Markdownのみの修正時はスキップ）**
5. **確認と報告**: 更新したファイルとローカルテストの実行結果を、ユーザーへの作業完了報告（walkthrough等）で明示する。**（Markdownのみの修正時は、ローカルテストの実行がスキップされた旨を報告に明記する）**
