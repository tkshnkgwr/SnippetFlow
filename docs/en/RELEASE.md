# Release Procedures (RELEASE.md)

This document defines the release procedures, version synchronization rules, and the operation of the automated packaging pipeline for "SnippetFlow" (Template Clipboard Manager).

---

## 1. Version Control and Automatic Synchronization Rules (SSOT)

This project mixes a frontend (React), Tauri (desktop integration), and an egui standalone version (Rust native). To prevent version divergence and inconsistency, please strictly adhere to the following rules.

### 1.1. Single Source of Truth (SSOT)
The source of truth for version information is **`package.json`** in the root directory. Version display in the frontend is loaded dynamically from `package.json`.

### 1.2. Files Requiring Manual Synchronization
Versions are not automatically synchronized between Node.js (`package.json`) and Rust (`Cargo.toml`). When releasing a new version, **be sure to manually update the following files to the same version**:

1. **`package.json`** (`"version"` field)
2. **`Cargo.toml` (root)** (`version` field)
3. **`src-tauri/Cargo.toml`** (`version` field)
4. **`README.md` / `README.ja.md`** version status badges at the top

> [!WARNING]
> Any version mismatch may cause build errors or failure in creating draft releases on GitHub Actions.

---

## 2. Automated Release Pipeline (CI/CD)

This project defines an automated build and packaging pipeline (`release.yml`) using GitHub Actions.

### 2.1. Trigger Conditions
Triggered when a Git tag in the format of **`v*`** (e.g., `v1.7.0`) is pushed to GitHub.

### 2.2. Artifacts Generated by the Pipeline
Build jobs run in parallel on Windows environment (`windows-latest`), and a draft GitHub Release is automatically created with the following artifacts uploaded:

1. **Tauri Version Installer**: Installers in `.msi` and `.exe` formats (built using WiX or NSIS, with support for Japanese language wizards).
2. **egui Standalone Version Binary**: `.zip` archive containing `Snippetflow.exe` compressed for Windows-x64.

---

## 3. Specific Release Steps

The steps to perform an actual release are as follows:

### Step 1: Quality Check and Pre-verification
Verify that the target commit passes local tests.
```bash
cargo fmt --check
cargo clippy --all-targets -- -D warnings
cargo test
```

### Step 2: Update Version Information
Update the `version` fields in `package.json`, `Cargo.toml`, and `src-tauri/Cargo.toml` to the new version (e.g., `1.8.0`), and commit the changes.

### Step 3: Organize CHANGELOG.md
Rename the `## [Unreleased]` section in `CHANGELOG.md` to the new version and date (e.g., `## [1.8.0] - YYYY-MM-DD`) and finalize the change log.

### Step 4: Create and Push Git Tag
Commit the version changes locally, tag them, and push them to GitHub.
```bash
# Commit changes
git add .
git commit -m "Bump version to v1.8.0"
git push origin main

# Create and push tag
git tag -a v1.8.0 -m "Release v1.8.0"
git push origin v1.8.0
```

### Step 5: Monitor Automated Build
Wait for the `Release Build` workflow to complete successfully (green status) in the GitHub `Actions` tab.

### Step 6: Verify and Publish Draft Release
1. Once the workflow is complete, check that a draft for `v1.8.0` has been generated on the GitHub Releases page.
2. Verify that all assets such as `Snippetflow.msi` / `Snippetflow_x64_en-US.msi` and `snippet_manager-windows-x64.zip` are present.
3. Write/format the release notes (copied from CHANGELOG) and click "Publish release" to publish the release.
